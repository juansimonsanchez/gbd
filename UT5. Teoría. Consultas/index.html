    1. INTRODUCCIÓN AL LENGUAJE DML

    A diferencia de los lenguajes de programación imperativos tradicionales (C, Java, Python), donde el desarrollador define el algoritmo paso a paso (cómo obtener los datos), SQL es un lenguaje declarativo. El usuario describe formalmente la estructura de los datos que desea recuperar (qué obtener), y es el Optimizador de Consultas de MySQL el encargado de calcular el plan de ejecución físico más eficiente para acceder al disco y la memoria.

    Para un Administrador de Sistemas, comprender el flujo interno de una consulta dentro de la instancia de MySQL es vital para diagnosticar latencias y cuellos de botella:

    [Conexión Cliente] ---> [Hebra de Conexión] ---> [Analizador Sintáctico/Semántico (Parser)]
    |
    [Resultados al Cliente] \<-- [Motor de Almacenamiento (InnoDB)] \<-- [Optimizador de Consultas]

    1. Parser & Preprocesador: Valida la sintaxis SQL de la consulta y verifica que las tablas y columnas existan en el diccionario de datos.

    2. Optimizador: Evalúa múltiples planes de acceso alternativos (uso de índices primarios, secundarios, escaneo de tablas completas) y calcula el coste estimado de I/O de CPU y disco para elegir el plan óptimo.

    3. Motor de Ejecución (Storage Engine): Solicita las páginas de datos requeridas al motor físico (generadamente InnoDB), aprovechando el Buffer Pool en memoria RAM para minimizar el acceso físico a disco.

    Uno de los errores conceptuales más frecuentes en estudiantes de administración de sistemas es asumir que MySQL procesa las cláusulas en el mismo orden en que se escriben en el editor. El orden lógico de procesamiento determina de forma rigurosa la visibilidad de los alias de columnas y el alcance de los filtros:

    Orden de Escritura Orden de Ejecución Lógico
    1. SELECT \<columnas> 1. FROM \<tablas> (Carga el origen de datos)
    2. FROM \<origen_datos> 2. ON \<condiciones_unión> (Filtra las composiciones)
    3. WHERE \<condición_filtro> 3. JOIN \<tablas_asociadas> (Instancia el producto cartesiano)
    4. GROUP BY \<atributos> 4. WHERE \<condiciones_tuplas> (Filtra filas individuales)
    5. HAVING \<condición_grupo> 5. GROUP BY \<atributos> (Segmenta en subgrupos)
    6. ORDER BY \<atributos> 6. HAVING \<condiciones_grupo> (Filtra los subgrupos creados)
    7. LIMIT \<desplaz., cantidad> 7. SELECT \<columnas> (Proyecta y calcula alias)
    8. DISTINCT (Elimina duplicados remanentes)
    9. ORDER BY \<atributos> (Ordena el conjunto final de salida)
    10. LIMIT (Recorta la porción de datos solicitada)

    [TABLE]

    2.2 La Cláusula SELECT

    2.2.1 Proyección Total (SELECT *)

    Aunque el uso del asterisco comodín (*) es ampliamente recurrente en entornos de desarrollo rápido, está estrictamente desaconsejado en entornos de producción debido a:

    1. Saturación del Tránsito de Red: Recuperar columnas innecesarias (especialmente campos TEXT o BLOB) desperdicia ancho de banda entre el servidor de base de datos y el servidor de aplicaciones.

    2. Invalidación de Índices de Cubrimiento (Covering Indexes): Si una consulta requiere solo dos columnas indexadas, y se utiliza *, MySQL se verá obligado a leer los bloques de datos de la clave primaria del disco (o el índice agrupado) en lugar de servir la petición utilizando únicamente el índice secundario en memoria RAM.

    2.3.4 Tratamiento de la Ausencia de Datos (IS NULL / IS NOT NULL)

    En el modelo relacional y SQL, NULL no equivale a un valor numérico cero, ni a una cadena de texto vacía "". Representa la "ausencia de valor" o "desconocimiento". Las comparaciones directas con = (como columna = NULL) siempre devolverán un resultado de tipo indeterminado (UNKNOWN) bajo la lógica triestatal de SQL. Por ende, para evaluar la presencia o ausencia de datos, deben emplearse de forma exclusiva los operadores estructurados IS NULL y IS NOT NULL.

    3. CONSULTAS SOBRE VARIAS TABLAS

    3.1 Concepto de Producto Cartesiano (Composición Cruzada o CROSS JOIN)

    Si intentamos consultar múltiples tablas en una sentencia separándolas por comas en el FROM, o invocando explícitamente CROSS JOIN sin especificar una condición de enlace, el motor ejecutará un producto cartesiano. Esto significa que asociará cada registro de la primera tabla con absolutamente todos los registros de la segunda tabla.

    Filas resultantes= Filas de Tabla A x Filas de Tabla B

    Si la Tabla A contiene 10000 registros y la Tabla B 5000, el producto cartesiano instanciará temporalmente en el servidor un conjunto de 50 millones de registros, saturando la memoria del proceso mysqld y pudiendo bloquear el servicio.

    1
2
3
    -- Producto cartesiano explícito (Ejecutar solo bajo estricta necesidad de diseño)

SELECT \* FROM fabricante CROSS JOIN producto;

    3.2 Composición Interna (INNER JOIN)

    La composición interna evalúa la intersección estricta de dos o más conjuntos. El motor devolverá únicamente aquellas tuplas que tengan una coincidencia real en ambas tablas basándose en la condición lógica de emparejamiento.

    Existen dos vías para implementar la unión física, siendo la explícita mediante la cláusula INNER JOIN ... ON la estándar y recomendada para preservar la legibilidad del código frente a filtros lógicos.

SELECT producto.nombre, fabricante.nombre AS fabricante

WHERE producto.codigo_fabricante = fabricante.codigo;

    5. SUBCONSULTAS (CONSULTAS ANIDADAS)

    Una subconsulta es una sentencia SELECT anidada dentro de otra consulta de nivel superior (llamada consulta externa). Las subconsultas permiten estructurar soluciones complejas sin requerir múltiples pasos o llamadas desde el lenguaje de programación de backend.

    5.1 Introducción y Tipología de Subconsultas

    Según el resultado que devuelve la consulta interna, las clasificamos en:

    1. Escalar: Devuelve una única fila con una única columna (un solo valor).

    2. De Fila: Devuelve una única fila con múltiples columnas asociadas.

    3. De Lista / Columna: Devuelve una columna con múltiples registros lógicos.

    4. De Tabla: Devuelve un conjunto bidimensional completo de filas y columnas (Derived Table).

    🔗 Documentación Oficial: Consulta las reglas de análisis semántico del anidamiento de sentencias DML en MySQL Manual - Subquery Syntax.

    6. ERRORES COMUNES Y GUÍA DE BUENAS PRÁCTICAS

    Para mantener un servidor MySQL con altas prestaciones bajo cargas de trabajo corporativas, el Administrador de Sistemas debe vigilar el comportamiento del DML:

    1. Evitar las composiciones cruzadas accidentales (CROSS JOIN implícitos): Validar siempre que cada INNER JOIN o LEFT JOIN cuente con su correspondiente cláusula restrictiva de coincidencia física (ON o USING).

    2. Utilizar filtros WHERE sobre columnas indexadas: Para evitar lecturas completas de tabla (Full Table Scans), asegúrate de que los filtros de búsqueda frecuente y las columnas que enlazan JOINs cuenten con un índice secundario asociado en la estructura DDL de la base de datos.

    3. No utilizar funciones directamente sobre columnas filtradas: Si consultas WHERE YEAR(fecha_registro) = 2026, MySQL se verá obligado a procesar la función matemática para cada fila de la tabla, invalidando los índices asociados a fecha_registro. En su lugar, utiliza rangos planos: WHERE fecha_registro BETWEEN '2026-01-01' AND '2026-12-31'.

    4. Validar las consultas analíticas en producción: Herramientas como EXPLAIN (analizadas a fondo en la Unidad 10) deben ser ejecutadas de forma sistemática ante cualquier consulta que devuelva latencias de respuesta superiores a los 100 ms en los logs de consultas lentas (slow_query_log).

CREATE TABLE departamento (

CREATE TABLE oficina (

codigo_oficina VARCHAR(10) PRIMARY KEY,

ciudad VARCHAR(50) NOT NULL,

telefono VARCHAR(20) NOT NULL,

superficie SMALLINT UNSIGNED NOT NULL CHECK (superficie \> 0) -- Espacio físico en metros cuadrados

) ENGINE=InnoDB;

-- 3. Creación de la tabla EMPLEADO

-- Integra fechas de contratación (DATE), horas de entrada (TIME) y estados lógicos (BOOLEAN)

CREATE TABLE empleado (

codigo_empleado INT PRIMARY KEY,

nombre VARCHAR(50) NOT NULL,

apellido1 VARCHAR(50) NOT NULL,

email VARCHAR(120) UNIQUE NOT NULL,

salario DECIMAL(10,2) NOT NULL,

FOREIGN KEY (codigo_departamento) REFERENCES departamento(codigo) ON UPDATE CASCADE ON DELETE SET NULL,

FOREIGN KEY (codigo_jefe) REFERENCES empleado(codigo_empleado) ON UPDATE CASCADE ON DELETE SET NULL

) ENGINE=InnoDB;

-- Combina lógica de fidelización VIP (BOOLEAN), precisión decimal ampliada y marcas temporales (DATETIME)

CREATE TABLE cliente (

codigo_empleado_rep_ventas INT,

) ENGINE=InnoDB;
